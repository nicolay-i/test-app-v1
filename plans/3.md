Ниже план **от текущего состояния `main`**, а не от желаемого состояния. Я бы зафиксировал это прямо: репозиторий сейчас всё ещё ближе к раннему slice, чем к полностью готовому lifecycle benchmark. README сам говорит про “first slice” и показывает только `init`, `preflight`, `run-matrix --dry-run`; `package.json` пока даёт базовые команды `bench`, `build`, `typecheck`, `lint`; `run-matrix` в коде всё ещё бросает ошибку при non-dry-run; `tasks/todomvc/task.yaml` сейчас содержит `evolution: []`; `runs/` содержит только `.gitkeep`, то есть proof artifacts пока не видны. ([GitHub][1])

# Главный принцип

Сейчас не надо добавлять новые модели, новые задачи или красивые отчёты. Нужно довести один trajectory до состояния, которому можно доверять:

```text
run-one
→ workspace
→ v0 generation
→ eval
→ artifacts
→ score
→ failure classification
→ report
→ v1..v4 evolution
→ cumulative regression tests
→ repair
→ trajectory summary
→ aggregation
```

Пока это не доказано на mock trajectory, real runs запускать рано.

# Фаза 3. Сделать failure classification и truthful scoring

**Цель:** score не должен врать, а failures должны быть пригодны для анализа.

Нужно избежать ситуации, где `1.00` означает “всё идеально”, хотя на деле прошли только текущие smoke checks. Visual capture отдельно нельзя называть visual similarity. Это особенно важно, потому что сейчас visual baseline comparison ещё не является доказанной частью benchmark.

Сделать `failure-summary.json`:

```json
{
  "status": "failed",
  "classification": "model_failure",
  "failed_phase": "e2e",
  "failed_checks": ["e2e"],
  "infra_suspected": false,
  "messages": [
    "E2E failed: completed filter did not show completed todos"
  ],
  "artifact_paths": {
    "install_log": "install.log",
    "build_log": "build.log",
    "playwright_report": "playwright-report",
    "screenshots": "screenshots"
  }
}
```

MVP-классификация:

```text
opencode_failure:
  OpenCode process failed before usable file changes.

infra_failure:
  pnpm install policy/network/cache/browser install failure.

model_failure:
  generated app does not build or product checks fail.

harness_failure:
  test runner/test assumptions look broken.

unknown:
  не удалось уверенно классифицировать.
```

В `score.json` разделить:

```json
{
  "visual_smoke_score": 1.0,
  "visual_similarity_score": null,
  "version_quality_smoke_score": 0.82,
  "benchmark_quality_score": null,
  "notes": [
    "visual_smoke_score means screenshot capture passed; no baseline similarity scoring is configured yet"
  ]
}
```

**DoD фазы:**

```text
- failed run всегда имеет failure-summary.json;
- visual_smoke не называется visual_similarity;
- score.md/report.md объясняет ограничения score;
- infra failures не считаются поражением модели по умолчанию.
```

# Минимальный Definition of Done на ближайший цикл

Ближайший цикл закончен, когда команда:

```bash
pnpm bench run-one \
  --task todomvc \
  --model deepseek-v4-flash-free \
  --system S2-maintainable-simple \
  --user U5-maintainable \
  --edit E2-smallest-maintainable-change \
  --versions 4 \
  --run-type mock
```

создаёт:

```text
runs/<trajectory-id>/
  trajectory-metadata.json
  trajectory-summary.json
  trajectory-report.md

  v0/
    prompt.md
    run-metadata.json
    opencode.stdout.log
    opencode.stderr.log
    opencode.events.jsonl
    git.diff
    check-results.json
    metrics.json
    score.json
    failure-summary.json
    report.md

  v1/
  v2/
  v3/
  v4/
```

После этого:

```bash
pnpm bench aggregate --config configs/mvp.yaml
```

создаёт:

```text
trajectory-results.jsonl
version-results.jsonl
scores.csv
leaderboard.md
```

и в `leaderboard.md` mock trajectory не попадает в ranking.

Только после этого стоит делать первый real sanity run на одной модели и двух prompt arms.
