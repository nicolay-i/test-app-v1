Ниже план, который я бы дал как рабочий roadmap для реализации. Главный принцип: **сначала сделать один честный `run-one`, потом lifecycle, потом matrix, потом ambiguity/negotiation**. Не надо сейчас расширять модели и задачи — это даст шум, а не знание.

# Цель ближайшего этапа

Довести проект до состояния, где один запуск:

```text
task + model + system prompt + user prompt + run
```

создаёт полностью воспроизводимый артефакт:

```text
workspace
prompt.md
opencode stdout/stderr/events
git diff
build logs
playwright logs
screenshots
check-results.json
metrics.json
score.json
failure-summary.json
run-metadata.json
```

И где сразу понятно:

```text
это mock или real run
это infra failure или model failure
visual — это capture или real similarity score
score — это smoke score или полноценный benchmark score
```

Пока это не сделано, matrix запускать рано.

# План по фазам

## Фаза 1. Стабилизировать `run-one` как атомарную единицу benchmark-а

**Приоритет: P0.**

Задача: один `run-one` должен быть самодостаточным, честным и пригодным для анализа без ручных шагов.

### 1.1. Добавить `run-metadata.json`

Файл:

```text
runs/<trajectory-id>/v0/run-metadata.json
```

Минимальный формат:

```json
{
  "schema_version": "0.1.0",
  "run_type": "mock",
  "task_id": "todomvc",
  "model_id": "deepseek-v4-flash-free",
  "provider_model": "opencode/deepseek-v4-flash-free",
  "system_prompt_id": "S2-maintainable-simple",
  "user_prompt_id": "U3-semantic-ui",
  "edit_prompt_id": "E2-smallest-maintainable-change",
  "run_number": 1,
  "version_id": "v0",
  "workspace_path": "...",
  "artifacts_path": "...",
  "prompt_path": "...",
  "started_at": "...",
  "completed_at": "...",
  "status": "passed | failed",
  "failure_classification": "none | infra_failure | model_failure | opencode_failure | harness_failure | unknown"
}
```

**Acceptance criteria:**

```text
- каждый run-one создаёт run-metadata.json
- metadata содержит run_type
- metadata содержит task/model/prompt ids
- metadata содержит paths к workspace/artifacts
- metadata содержит final status
```

---

### 1.2. Жёстко развести `mock` и `real`

Сейчас mock полезен для pipeline, но опасен для leaderboard.

Добавить в CLI явную модель:

```bash
pnpm bench run-one ... --run-type mock
pnpm bench run-one ... --run-type real
```

Или оставить:

```bash
--mock-opencode
```

но внутри всегда писать:

```json
"run_type": "mock"
```

В `score.json` тоже добавить:

```json
{
  "run_type": "mock",
  "eligible_for_leaderboard": false
}
```

Для real:

```json
{
  "run_type": "real",
  "eligible_for_leaderboard": true
}
```

**Acceptance criteria:**

```text
- mock score не может случайно попасть в real leaderboard
- report по умолчанию исключает mock
- mock явно виден в filename/report/metadata
```

---

### 1.3. Передавать prompt в OpenCode через файл

Сейчас длинные prompts лучше не передавать как positional argument. Нужно сохранять prompt:

```text
runs/<trajectory-id>/v0/prompt.md
```

И вызывать OpenCode примерно так:

```bash
opencode run \
  --model opencode/deepseek-v4-flash-free \
  --dir <workspace> \
  --format json \
  --auto \
  --file <artifacts>/prompt.md \
  "Read the attached benchmark prompt and implement the requested app."
```

**Acceptance criteria:**

```text
- полный prompt всегда сохраняется как artifact
- OpenCode получает prompt через --file
- stdout/stderr/events сохраняются рядом
- adapter не ломается на длинных prompts
```

---

### 1.4. Добавить `failure-summary.json`

После каждого evaluator run писать:

```text
runs/<trajectory-id>/v0/failure-summary.json
```

Пример:

```json
{
  "status": "failed",
  "classification": "model_failure",
  "failed_phase": "e2e",
  "failed_checks": ["e2e"],
  "infra_suspected": false,
  "messages": [
    "E2E failed: completed filter did not show completed todos"
  ],
  "artifact_paths": {
    "install_log": "install.log",
    "build_log": "build.log",
    "playwright_report": "playwright-report",
    "screenshots": "screenshots"
  }
}
```

Первые правила классификации можно сделать простыми:

```text
pnpm install failed due to policy/network/cache → infra_failure
opencode command failed before file changes → opencode_failure
build failed after generated code → model_failure
playwright browser missing → infra_failure
e2e assertion failed → model_failure_or_harness_failure
visual screenshot failed due to page crash → model_failure
unknown error → unknown
```

**Acceptance criteria:**

```text
- любой failed run имеет failure-summary.json
- summary показывает failed phase
- summary показывает предполагаемый failure class
- summary даёт ссылки на logs/artifacts
```

---

### 1.5. Переименовать visual score

До baseline comparison нельзя называть это visual similarity.

Сейчас правильнее:

```json
{
  "visual_smoke_score": 1.0,
  "visual_similarity_score": null
}
```

И в `score.json` добавить notes:

```json
{
  "notes": [
    "visual_smoke_score means screenshot capture passed; no baseline similarity scoring is configured yet"
  ]
}
```

**Acceptance criteria:**

```text
- visual capture не выдаётся за visual similarity
- score report явно различает visual_smoke и visual_similarity
```

### 1.6. Добавить capture console/page errors

В Playwright checks сохранять:

```text
console-errors.json
page-errors.json
```

Даже если тесты прошли, runtime errors должны быть видны.

**Acceptance criteria:**


## Фаза 3. Добавить TodoMVC evolution steps

**Приоритет: P1.**

После честного v0 добавить lifecycle. Не начинать с 10 изменений. Начать с 4.

В `tasks/todomvc/task.yaml` добавить:

```yaml
evolution:
  - id: 01-add-due-dates
    prompt: evolution/01-add-due-dates.md
    tests:
      - tests/evolution/01-due-dates.spec.ts

  - id: 02-add-search
    prompt: evolution/02-add-search.md
    tests:
      - tests/evolution/02-search.spec.ts

  - id: 03-add-tags
    prompt: evolution/03-add-tags.md
    tests:
      - tests/evolution/03-tags.spec.ts

  - id: 04-remove-tags
    prompt: evolution/04-remove-tags.md
    tests:
      - tests/evolution/04-remove-tags.spec.ts
```

Я бы взял именно `remove-tags` как четвёртый шаг. Add-only lifecycle слишком мягкий. Удаление фичи быстро показывает мёртвый код, копипасту и плохую модель данных.

### Evolution prompts

`01-add-due-dates.md`:

```text
Add optional due dates to todos.

Requirements:
- User can create a todo without a due date.
- User can assign a due date while editing a todo.
- Due date is visible on the todo row.
- Overdue active todos are visually distinguishable.
- Due dates persist after refresh.
- Existing TodoMVC behavior must keep working.
```

`02-add-search.md`:

```text
Add search by todo title.

Requirements:
- Search filters visible todos by title.
- Search works together with All / Active / Completed filters.
- Clearing search restores the current filter view.
- Search does not delete or mutate todos.
- Existing due date behavior must keep working.
```

`03-add-tags.md`:

```text
Add tags to todos.

Requirements:
- Todo can have zero or more tags.
- Use predefined tags: Work, Personal, Urgent.
- User can edit tags for an existing todo.
- User can filter by one tag at a time.
- Tags persist after refresh.
- Existing search, due dates, and TodoMVC behavior must keep working.
```

`04-remove-tags.md`:

```text
Remove the tags feature completely.

Requirements:
- Remove tag UI from create/edit flows.
- Remove tag filter UI.
- Existing todos should still load even if old persisted data contains tags.
- Search, due dates, filters, editing, completion, deletion, and persistence must keep working.
- Remove dead tag-related code where practical.
```

**Acceptance criteria:**

```text
- task.yaml has at least 4 evolution steps
- each step has prompt file
- each step has Playwright tests
- previous tests continue running as regression tests
```

## Фаза 10. Requirements negotiation benchmark

**Приоритет: P3.**

Это важный слой, но его не надо смешивать с v0/lifecycle до стабилизации.

Добавить структуру:

```text
tasks/todomvc/negotiation/
  01-feature-already-exists.yaml
  02-user-misconception-active-filter.yaml
  03-underspecified-tags.yaml
  04-hidden-migration-projects.yaml
```

Формат scenario:

```yaml
id: 03-underspecified-tags
version: v2
request: |
  Add tags to todos.

expectedDecision: clarify_or_assume_with_default

requiredFindings:
  - tags affect the todo data model
  - tags affect create/edit UI
  - tags affect persistence
  - filtering behavior is unspecified

requiredQuestions:
  - single vs multiple tags
  - predefined vs custom tags
  - whether filtering by tag is required

prohibitedBehavior:
  - silently add decorative tags only
  - duplicate todo state model
  - hard-code tag logic separately in multiple views

oracleAnswer: |
  Todos can have zero or more tags.
  Use predefined tags: Work, Personal, Urgent.
  Users can filter by one tag at a time.
```

Добавить команду:

```bash
pnpm bench negotiate-one \
  --task todomvc \
  --scenario 03-underspecified-tags \
  --model deepseek-v4-flash-free \
  --system S2-maintainable-simple \
  --run-type mock
```

Preflight protocol:

```json
{
  "decision": "proceed | clarify | already_exists | conflict | out_of_scope",
  "confidence": 0.0,
  "reason": "",
  "questions": [],
  "recommended_default": "",
  "affected_areas": [],
  "will_edit": false
}
```

**Acceptance criteria:**

```text
- agent can be forced into preflight-only mode
- decision JSON is parsed and scored
- oracle answer can be sent after clarification
- final implementation is evaluated normally
```
