# План этапа 6: рабочий прототип наблюдаемых агентных траекторий

## 1. Контекст и реальная задача

Проект должен стать сервисом для наблюдения за тем, как агент развивает код при
разных промптах и вводных:

```text
исходная задача
-> реализация v0
-> последовательные изменения v1, v2, ...
-> проверки и регрессии
-> технические продолжения
-> repairs
-> уточнения требований
-> возможное участие человека
```

Объект исследования - не только финальный код, а вся траектория его получения.
Нужно видеть:

```text
- что агент смог реализовать;
- на каком этапе появились проблемы;
- как между этапами менялся расход токенов;
- как росли объем кода, сложность, дублирование и размер изменений;
- сколько потребовалось автоматических continuations и repairs;
- когда агенту не хватило исходных требований;
- какой объем дополнительной информации потребовался;
- где реально вмешивался человек.
```

Это не поиск единственного лучшего промпта и не статистическое доказательство
превосходства одного варианта. Ближайший результат - рабочий исследовательский
прототип, который надежно собирает и показывает эти данные.

Предлагаемое название milestone:

```text
R2 Prototype - Observable Agent Lifecycle
```

---

## 2. Граница прототипа

Прототип должен получить минимум по одной успешной реальной траектории `v0->v2`
для:

```text
U3-semantic-ui
U5-maintainable
```

при одинаковых:

```text
task
model
system prompt
edit prompt
наборе версий
тестах
лимитах continuations
лимитах repairs
clarification protocol
```

Прототип проверяет:

```text
- корректность lifecycle runner-а;
- наблюдаемость расхода ресурсов и feedback loops;
- возможность сравнить две реальные траектории;
- возможность учесть требуемый надзор и реальные действия человека.
```

Прототип пока не отвечает статистически на вопрос, с какой скоростью код в общем
случае становится неподдерживаемым. Для такого вывода позже потребуются:

```text
- более длинный lifecycle;
- несколько успешных trajectories на arm;
- несколько задач и моделей;
- отдельная система анализа поддерживаемости;
- анализ variance и повторяемости.
```

---

## 3. Что считаем сигналами деградации

На этом этапе не вводится порог, после которого код объявляется
"неподдерживаемым", и не рассчитывается общий maintainability score.

Между версиями и независимыми запусками сравниваются:

```text
token usage по этапам;
LOC и прирост LOC;
largest file LOC;
files touched;
lines added/deleted;
rewrite ratio;
duplicate ratio;
cyclomatic complexity;
dependency cycles;
появление регрессий;
число repairs до успешной версии;
необходимость дополнительных уточнений.
```

Code-health metrics являются диагностическими proxy. После рабочего прототипа к
собранным артефактам будет подключена отдельная система анализа
поддерживаемости.

Пока разрешен вывод только такого уровня:

```text
В наблюдаемых успешных траекториях U3 и U5 различались расход токенов,
объем изменений, code-health metrics и необходимый feedback loop.
```

Нельзя утверждать:

```text
U3 доказанно лучше U5.
Код стал неподдерживаемым на версии vN.
Наблюдаемое различие вызвано только вариантом промпта.
```

## 4. Единицы наблюдения

### 4.1. Trajectory

Одна независимая история развития приложения:

```text
v0 base TodoMVC
-> v1 due dates
-> v2 search
```

Новая trajectory всегда начинается с чистого scaffold и не использует workspace
другой trajectory.

### 4.2. Version

Один этап lifecycle. Версия может включать:

```text
requirements preflight;
clarification rounds;
initial implementation call;
continuation calls;
evaluation;
repair calls;
повторные evaluations.
```

### 4.3. OpenCode attempt

Один запуск `opencode run`. Для прототипа не нужно вводить отдельную
универсальную доменную модель `AgentCall`: достаточно расширить существующие
`opencode-attempts.json`, version summary и trajectory summary.

Для каждого attempt хранить:

```text
attempt number;
purpose: preflight | implementation | continuation | clarification | repair;
session id;
continued session id;
prompt path;
start/end timestamps;
exit status;
failure classification;
usage;
raw JSONL;
assistant response.
```

---

## 5. Правило выборки и остановки

Для каждого prompt arm:

```text
- запускать независимые trajectories до первого успешного case;
- не запускать больше трех trajectories;
- для подробного сравнения использовать первый успешный case;
- не выбирать лучший case постфактум;
- сохранить и перечислить все предшествующие failures;
- если за три trajectories успеха нет, arm остается без успешного case.
```

Trajectory считается успешной, если она дошла до `v2` и cumulative suite прошел
после разрешенных continuations, clarifications и repairs.

Continuation не является новой trajectory. Repair не является новой trajectory.

## 6. Типы feedback loop

### 6.1. Continuation

Continuation используется только при техническом прерывании OpenCode-вызова:

```text
provider error;
network/external service failure;
timeout;
процесс OpenCode был прерван;
поток завершился без корректного terminal event.
```

Continuation не используется, если агент завершил работу, но evaluator нашел
ошибки. Это repair.

Правила continuation:

```text
- продолжать ту же OpenCode session, если session id доступен;
- отправлять сообщение "Продолжай выполнение текущей задачи с того места, где остановился.";
- не повторять исходный benchmark prompt;
- не откатывать workspace;
- сохранять JSONL и usage каждого attempt отдельно;
- ограничивать число continuations конфигурацией;
- при недоступной session разрешать fallback в новую session того же workspace;
- явно помечать fallback, чтобы он не выглядел как same-session continuation.
```

Пример:

```bash
opencode run --session <session-id> \
  "Продолжай выполнение текущей задачи с того места, где остановился."
```

### 6.2. Repair

Repair выполняется после завершившегося implementation/continuation call, если
evaluator нашел функциональные, сборочные или проверяемые quality-проблемы.

```text
implementation
-> evaluation failed
-> repair 1
-> evaluation failed
-> repair 2
-> evaluation passed или repair limit reached
```

Автоматический repair является дополнительным feedback loop, но не считается
человеческим вмешательством.

### 6.3. Clarification

Clarification возникает, когда до реализации агент сообщает, что важных данных
недостаточно или требования противоречивы.

Clarification отличается от repair:

```text
clarification - агенту не хватает решения по требованиям;
repair - реализация уже проверена и не прошла evaluator.
```

### 6.4. Human intervention

Фактическими действиями человека считаются:

```text
human_answer;
human_prompt_correction;
human_acceptance_correction;
human_code_edit.
```

Автоматический continuation, oracle answer и автоматический repair не считаются
фактическими действиями человека.

## 7. Таблица решений runner-а

| Событие | Действие | Учет |
|---|---|---|
| Provider/network error | Continuation | automatic continuation |
| Timeout или interrupted process | Continuation | automatic continuation |
| Session недоступна | New-session fallback в том же workspace | continuation fallback |
| Agent call завершен, evaluation failed | Repair | automatic repair |
| Agent запросил существенное уточнение | Получить oracle/human answer | required supervision |
| Harness/configuration failure | Остановить trajectory | harness failure |
| Repair limit исчерпан | Остановить trajectory | terminal model failure |
| Человек изменил код | Продолжить только с явной фиксацией diff | human code edit |

Классификация должна быть machine-readable и не определяться только по ненулевому
exit code.

## 8. Clarification protocol

### 8.1. Двухфазный этап

Каждая версия выполняется в двух фазах:

```text
1. Requirements preflight без редактирования workspace.
2. Implementation после решения proceed или после получения ответа.
```

Preflight возвращает структурированное решение:

```json
{
  "decision": "proceed | clarify | conflict | already_exists",
  "reason": "",
  "questions": [],
  "affected_areas": []
}
```

Если получено `proceed`, runner запускает implementation.

Если получено `clarify`, runner:

1. Сохраняет вопросы.
2. Выбирает доступный источник ответа.
3. Сохраняет ответ и его источник.
4. Передает исходный запрос и ответ в implementation phase.
5. Учитывает отдельный usage preflight и clarification continuation.

Preflight должен одинаково применяться к U3 и U5 и не должен менять workspace.

### 8.2. Источники ответа

```text
oracle_answer          - заранее зафиксированный ответ benchmark-а;
scenario_answer        - ответ из контролируемого negotiation scenario;
human_answer           - фактический ответ человека.
```

Oracle/scenario answer позволяет выполнить воспроизводимый запуск. Он не является
реальным действием человека, но показывает, что без дополнительной информации
агент не продолжил бы реализацию.

### 8.3. Несколько clarification rounds

Версия может потребовать более одного уточнения. Число rounds ограничивается
`maxClarificationRounds`.

```text
preflight
-> clarify 1
-> answer 1
-> clarification continuation
-> clarify 2
-> answer 2
-> proceed или clarification limit reached
```

Если после лимита агент все еще не готов продолжать, версия получает terminal
status `clarification_limit_reached`.

### 8.4. Артефакты clarification

```text
v1/
  preflight/
  clarification-1/
    request.json
    question.md
    answer.md
    result.json
  clarification-2/
  opencode-attempts/
  repair-1/
  version-summary.json
```

Артефакты initial implementation остаются в корне версии по текущей схеме.
Переносить их в новый каталог `implementation/` для прототипа не требуется.

Можно переиспользовать существующие structured decision и oracle-механику из
`packages/runner/src/negotiation.ts`, но clarification должен стать частью
обычной trajectory, а не только отдельной negotiation-командой.

## 9. Учет участия человека

Нужно разделять два разных результата.

### 9.1. Required supervision

Показывает, сколько дополнительной информации потребовалось агенту независимо от
того, ответил человек или автоматический oracle:

```text
versions_requiring_clarification;
clarification_rounds;
questions_total;
answer_words;
answer_tokens;
clarification_limit_reached.
```

### 9.2. Actual human activity

Показывает только реальные действия человека:

```text
human_answers_total;
human_answer_words;
human_prompt_corrections;
human_acceptance_corrections;
human_code_edits;
manual_files_changed;
manual_lines_added;
manual_lines_deleted.
```

Wall-clock ожидание ответа не считается человеческим трудом. Позже можно добавить
явно фиксируемое `active_minutes`, но runner не должен выводить его из времени
ожидания.

### 9.3. Хранение данных

Для первого прототипа отдельный общий `interventions.jsonl` не обязателен.
Источник истины:

```text
clarification-N artifacts;
repair-N artifacts;
opencode-attempts.json;
version-summary.json;
trajectory-summary.json.
```

Version summary хранит список feedback events, trajectory summary - агрегированные
счетчики. Полные вопросы, ответы и prompts остаются отдельными файлами.

## 10. Несколько repairs

Текущая реализация жестко создает только `repair-1`. Нужен цикл:

```text
for attempt in 1..maxRepairAttempts
  compile repair prompt из последнего failure summary
  run agent
  evaluate
  save repair-N artifacts
  stop immediately on pass
```

Каждый следующий repair получает диагностику предыдущей оценки, а не исходную
ошибку.

Структура:

```text
v1/
  opencode-attempts/
  repair-1/
  repair-2/
  repair-3/
  version-summary.json
```

Version/trajectory summaries показывают:

```text
repairs attempted;
repairs until success;
versions passed without repair;
versions passed after repair;
versions failed after repair limit;
repair usage;
repair latency.
```

## 11. Token usage

### 11.1. Проверенная схема OpenCode

На реальном запуске OpenCode `1.17.18` usage приходит в событиях
`step_finish.part.tokens`:

```json
{
  "type": "step_finish",
  "part": {
    "tokens": {
      "total": 9431,
      "input": 217,
      "output": 246,
      "reasoning": 8,
      "cache": {
        "read": 8960,
        "write": 0
      }
    },
    "cost": 0
  }
}
```

Для каждого проверенного model step выполняется:

```text
total = input + output + reasoning + cache.read + cache.write
```

Сумма 18 `step_finish` одной реальной сессии совпала с агрегатами OpenCode:

```text
input:        16 116
output:        4 593
reasoning:     1 297
cache read:  277 376
cache write:       0
total:       299 382
```

Следовательно, `step_finish` содержит расход отдельного model step, а не
накопительный итог сессии. События внутри одного OpenCode call нужно суммировать.

### 11.2. Правила агрегации

```text
step_finish -> OpenCode attempt
OpenCode attempts -> phase
preflight/implementation/continuations/repairs -> version
versions -> trajectory
```

Требования:

```text
- разбирать tokens только из подходящих step/usage events;
- дедуплицировать события по part.id;
- хранить input/output/reasoning/cache read/cache write отдельно;
- считать total вместе с cache usage;
- не считать cost: 0 отсутствием данных;
- при неполных данных использовать partial;
- сохранять raw JSONL для аудита;
- не суммировать повторно cumulative session snapshots.
```

Для same-session continuation:

```text
- JSONL каждого нового вызова учитывается как delta;
- если fallback берет session aggregate, используется разница before/after;
- одинаковые part.id не учитываются дважды.
```

### 11.3. Метрики

```text
preflight tokens;
clarification tokens;
implementation tokens;
continuation tokens;
repair tokens;
version total tokens;
lifecycle total tokens;
tokens per passed version;
repair token ratio;
cache read ratio;
reported cost.
```

Для сравнения роста затрат между этапами отдельно показывать:

```text
vN total / vN-1 total;
absolute token delta;
non-cache token delta;
cache read delta.
```

## 12. Проверки и упрощенный fairness для `v0->v2`

### 12.1. Cumulative suite

#### v0

```text
создание задач;
completion;
filters;
persistence;
semantic UI.
```

#### v1

```text
все проверки v0;
создание с due date и без нее;
видимость due date;
overdue behavior;
persistence due date.
```

#### v2

```text
все проверки v0/v1;
search by title;
search без изменения данных;
search совместно с Active/Completed;
очистка search;
сохранение due date и persistence behavior.
```

### 12.2. Минимальный fairness scope

Для прототипа достаточно:

```text
одна корректная контрольная реализация v0->v2;
одна mutation, ломающая due dates;
одна mutation, ломающая search;
подтвержденный cumulative запуск suite;
coverage map critical requirements -> tests;
аудит selectors на отсутствие зависимости от class names/component structure.
```

Вторая альтернативная корректная реализация переносится в следующий milestone.

Fixtures для tags и remove-tags пока не нужны.

## 13. Минимальный эксперимент U3 и U5

### 13.1. Дизайн

```text
Task: TodoMVC
Model: один фиксированный OpenCode model
System: S2-maintainable-simple
Edit: E2-smallest-maintainable-change
Arm A: U3-semantic-ui
Arm B: U5-maintainable
Lifecycle: v0->v2
Maximum trajectories: 3 на arm
Selected case: первый успешный case на arm
```

Порядок arm/run фиксируется seed до начала запусков. Runner не меняется между
сравниваемыми cases.

Для каждого arm показать:

```text
attempted trajectories;
successful trajectories;
first successful trajectory;
infra failures;
model failures;
harness failures;
continuations;
repairs;
required clarifications;
actual human interventions.
```

Если хотя бы один arm не получил успешный case за три trajectories, прототип
runner-а может быть технически готов, но сравнительный эксперимент считается
незавершенным.

## 15. Reproducibility baseline

Перед реальными сравниваемыми запусками:

1. Завершить deterministic proof suite.
2. Выполнить typecheck и проверки runner-а.
3. Создать clean committed baseline.
4. Не менять runner во время эксперимента.
5. Связать каждый execution с runner commit и manifest.

Обязательные команды:

```bash
pnpm typecheck
pnpm proof:mock
pnpm proof:fairness
pnpm bench verify-opencode-parser
pnpm bench verify-opencode-retry
```

После реализации clarification и multiple repairs добавить deterministic fixtures
для этих lifecycle branches.

Для сравниваемых real executions:

```text
repo_dirty = false;
eligible_for_published_results = true;
tested_runner_commit одинаковый;
artifact manifest verified;
порядок запусков зафиксирован seed;
```

## 17. Definition of Done

### 17.1. Runner ready

```text
1. OpenCode usage извлекается из реальных step_finish events.
2. Usage агрегируется без дублирования по attempts, phases, versions и trajectory.
3. Input/output/reasoning/cache usage сохраняются отдельно.
4. Continuation используется только для технических прерываний.
5. Continuation продолжает существующую session, когда это возможно.
6. Workspace не откатывается перед continuation.
7. maxRepairAttempts реально поддерживает repair-N.
```
