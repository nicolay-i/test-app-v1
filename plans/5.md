# План следующего этапа: от engineering MVP к исследовательски надёжному benchmark

Текущий harness уже умеет проводить реальные `v0→v2` trajectories, repairs, aggregation, negotiation preflight и blind jury export. Следующий этап должен не расширять количество функций, а повысить **достоверность результатов**.

Главная цель этапа:

> Любой опубликованный результат должен быть однозначно привязан к конкретному коду runner-а, task/prompt definitions и raw artifacts; lifecycle cost должен измеряться; score не должен выдавать smoke-проверки за полную оценку качества.

Предлагаемое название milestone:

```text
R2 — Reproducible Lifecycle Benchmark
```

# Порядок работ

## Фаза 1. Зафиксировать воспроизводимость proof-записей

**Приоритет: P0.**

Сейчас главный методологический риск — proof мог быть получен на dirty working tree, а `source_commit` не описывает незакоммиченные изменения.

### 1.1. Расширить execution manifest

Добавить:

```json
{
  "source_commit": "d2f158b...",
  "repo_dirty": false,
  "runner_source_hash": "...",
  "package_json_hash": "...",
  "lockfile_hash": "...",
  "config_hash": "...",
  "task_hashes": {},
  "prompt_hashes": {},
  "scaffold_hash": "...",
  "working_tree_patch_sha256": null
}
```

`runner_source_hash` должен включать как минимум:

```text
packages/runner/**
package.json
pnpm-lock.yaml
tsconfig*
configs/**
tasks/<active-task>/**
prompts/**
scaffolds/<active-scaffold>/**
```

### 1.2. Ввести policy для dirty tree

По умолчанию real benchmark и committed proof должны требовать чистое дерево:

```bash
pnpm bench run-one ... --run-type real
```

При dirty tree:

```text
Refusing real benchmark run: repository has uncommitted changes.
Use --allow-dirty only for debugging.
```

Для диагностического запуска:

```bash
--allow-dirty
```

runner должен сохранить:

```text
source.patch
source.patch.sha256
```

и отметить:

```json
{
  "reproducible_from_commit_only": false,
  "eligible_for_published_results": false
}
```

### 1.3. Усилить compact proof records

Новый формат:

```json
{
  "proof_schema_version": "0.2.0",
  "tested_runner_commit": "d2f158b...",
  "repo_dirty": false,
  "execution_id": "...",
  "command": "...",
  "exit_code": 0,
  "execution_manifest_sha256": "...",
  "artifact_manifest_sha256": "...",
  "verified_files": 137,
  "run_type": "real",
  "trajectory_status": "passed",
  "created_at": "..."
}
```

### Definition of Done

```text
- real proof нельзя создать на dirty tree без явного override;
- proof связан с execution manifest и artifact manifest через SHA-256;
- по proof видно, какой именно код runner-а запускался;
- verify-run проверяет эти связи;
- старые proof-файлы помечены legacy или заменены.
```

# Фаза 3. Реализовать lifecycle usage и cost

**Приоритет: P0/P1.**

Это главный функциональный пробел относительно исходной цели проекта.

Нужно получить:

```text
initial generation tokens
edit tokens per version
repair tokens
total lifecycle tokens
reported cost, если доступен
tokens per passing version
```

## 3.1. Нормализовать usage model

```ts
type UsageStatus =
  | "complete"
  | "partial"
  | "unavailable"
  | "invalid";

type AgentUsage = {
  status: UsageStatus;
  inputTokens: number | null;
  outputTokens: number | null;
  reasoningTokens: number | null;
  cacheReadTokens: number | null;
  cacheWriteTokens: number | null;
  totalTokens: number | null;
  reportedCost: number | null;
  currency: string | null;
  source: "events" | "session_export" | "stats" | "unavailable";
};
```

Не заменять отсутствие данных нулём.

Плохо:

```json
"total_tokens": 0
```

Правильно:

```json
{
  "total_tokens": null,
  "usage_status": "unavailable"
}
```

## 3.2. Сохранять usage по каждой попытке

На уровне версии:

```json
{
  "generation_usage": {},
  "repair_usage": [],
  "version_total_usage": {}
}
```

На уровне trajectory:

```json
{
  "lifecycle_tokens": 145320,
  "lifecycle_reported_cost": 0.0,
  "usage_complete": true,
  "tokens_per_passing_version": 48440,
  "repair_token_ratio": 0.18
}
```

## 3.3. Best-effort fallback

Последовательность источников:

```text
1. usage из OpenCode JSONL events;
2. usage из session export/stats, если доступно;
3. unavailable с явной причиной.
```

Нельзя приблизительно токенизировать текст локальным tokenizer-ом и выдавать это за provider billing usage. При необходимости можно хранить дополнительное поле:

```text
estimated_prompt_tokens
```

но не смешивать его с reported usage.

## 3.4. Метрики lifecycle efficiency

Минимальный набор:

```text
total lifecycle tokens
tokens per attempted version
tokens per passing version
repair tokens
repair token ratio
successful versions per 100k tokens
quality-adjusted survival per 100k tokens
```

Пример:

```text
successful_versions_per_100k_tokens =
passed_versions / lifecycle_tokens * 100000
```

### Definition of Done

```text
- usage сохраняется для v0, edits и repairs;
- отсутствующие токены представлены null + status;
- trajectory summary агрегирует usage;
- efficiency leaderboard исключает trajectories с unavailable usage либо показывает их отдельно;
- proof содержит хотя бы один real run с complete или честно partial usage.
```

# Фаза 4. Сделать scoring честным и расширяемым

**Приоритет: P1.**

Сейчас score полезен как smoke indicator, но его нельзя использовать как полноценную оценку приложения.

## 4.1. Переименовать текущий итог

Вместо:

```text
version_quality
```

использовать:

```text
version_quality_smoke_score
```

А новые поля оставить такими:

```json
{
  "version_quality_smoke_score": 1.0,
  "benchmark_quality_score": null,
  "visual_similarity_score": null,
  "jury_quality_score": null
}
```

Это исключит ошибочную интерпретацию `1.00`.

## 4.2. Добавить deterministic requirement adherence

Для каждой версии task должен содержать checklist:

```yaml
requirements:
  - id: create-todo
    type: e2e
    weight: 1
    critical: true

  - id: persistence
    type: value
    weight: 1
    critical: true

  - id: overdue-visible
    type: observable-ui
    weight: 0.5
    critical: false
```

Результат:

```json
{
  "requirement_adherence_score": 0.88,
  "requirements_passed": 14,
  "requirements_total": 16,
  "critical_requirements_failed": []
}
```

Нельзя использовать build success как proxy для prompt adherence.

## 4.3. Добавить code-health metrics

Первая итерация:

```text
duplicate code ratio
largest file LOC
largest file growth
total LOC growth
files touched per change
rewrite ratio
dependency count changes
dependency cycles
unused exports
cyclomatic complexity outliers
```

Инструменты:

```text
jscpd
dependency-cruiser
ESLint complexity
ts-prune или современный аналог
git diff parser
```

Хранить raw metrics и score отдельно:

```json
{
  "raw": {
    "duplicate_ratio": 0.04,
    "dependency_cycles": 0,
    "largest_file_loc": 430
  },
  "normalized": {
    "duplication_score": 0.92,
    "dependency_health_score": 1.0
  }
}
```

## 4.4. Не добавлять LLM judge первым

Сначала:

```text
deterministic adherence
code-health
regression
cost
```

LLM/jury можно добавить позже как дополнительный слой, а не источник истины.

### Definition of Done

```text
- smoke score явно назван smoke;
- prompt adherence больше не равен build score;
- requirement checklist имеет machine-readable output;
- code-health содержит как минимум duplication, complexity и dependency metrics;
- benchmark_quality_score остаётся null до утверждения формулы.
```

# Фаза 5. Завершить fairness audit тестов

**Приоритет: P1.**

Документ с правилами уже есть. Теперь нужно доказать их экспериментально.

## 5.1. Создать три fixtures

```text
tasks/todomvc/test-fixtures/
  compliant-reference-a/
  compliant-alternative-dom-b/
  intentionally-broken-c/
```

### A — текущая корректная реализация

Стандартная структура.

### B — корректная альтернативная реализация

Изменить:

```text
DOM hierarchy
component structure
state organization
localStorage key
class names
```

При этом сохранить пользовательское поведение.

### C — намеренно сломанная

Например:

```text
completed filter показывает все задачи
```

или:

```text
localStorage не восстанавливается
```

## 5.2. Требуемый результат

```text
A passes
B passes
C fails ровно на связанном criterion
```

Если B падает, тест implementation-specific. Если C проходит, тест слишком слабый.

## 5.3. Связать тесты с требованиями

Создать:

```text
tasks/todomvc/tests/coverage-map.yaml
```

```yaml
tests:
  base/e2e.spec.ts:create-todo:
    requirements:
      - create-todo
    observableBehavior: true
    implementationAssumptions: []

  base/values.spec.ts:persistence:
    requirements:
      - persistence
    observableBehavior: true
    implementationAssumptions: []
```

### Definition of Done

```text
- есть альтернативная корректная реализация;
- есть намеренно сломанная реализация;
- suite проходит на A и B;
- suite падает на C;
- каждый critical requirement покрыт хотя бы одним тестом;
- спорные selectors документированы или заменены.
```

# Фаза 7. Провести первый повторяемый A/B эксперимент

**Приоритет: P1/P2.**

Текущий `n=1` — только sanity. Следующий эксперимент должен начать проверять повторяемость.

## Дизайн

```text
Task: TodoMVC
Model: DeepSeek V4 Flash
System prompt: фиксированный S2
Edit prompt: фиксированный E2
User arms:
  A = U3 semantic UI
  B = U5 maintainable
Runs per arm: 3
Versions: v0→v2
Temperature/sampling: фиксировать, если провайдер позволяет
Execution order: чередовать или рандомизировать
```

Всего:

```text
2 arms × 3 runs × 3 versions = 18 version attempts
```

Repairs считаются отдельно.

## Необходимые результаты

По каждому arm:

```text
v0 pass rate
v0→v2 survival rate
mean passed versions
regression-free rate
repair-free rate
mean smoke quality
requirement adherence
lifecycle tokens
tokens per passing version
code-health growth
variance
```

## Порядок запуска

Не запускать сначала три A, потом три B. Лучше:

```text
A1
B1
B2
A2
A3
B3
```

или использовать seeded random ordering. Это уменьшает влияние временных provider changes и rate conditions.

## Выводы

Разрешённая формулировка:

```text
В этой небольшой выборке U3 имел 3/3 survival, U5 — 2/3.
```

Неразрешённая формулировка:

```text
U3 доказанно лучше U5.
```

### Definition of Done

```text
- минимум 3 независимых runs на arm;
- порядок запусков контролируется;
- raw artifacts сохранены;
- infra failures анализируются отдельно;
- вывод включает variance и размер выборки;
- mock результаты не участвуют.
```

# Фаза 9. Усилить negotiation benchmark

**Приоритет: P2.**

Текущий proof уже показал, что harness умеет обнаруживать protocol violation. Теперь нужно проверить полноценное понимание текущего приложения.

## 9.1. Preflight на snapshot конкретной версии

Вход должен содержать не только общую фразу про TodoMVC, а:

```text
current version
applied evolution steps
current requirement checklist
current test inventory
known features
compact file tree
current failures
```

## 9.2. Физически отделить preflight

```text
source workspace
→ ephemeral preflight copy
→ agent analysis
→ git diff/protocol check
→ discard copy
```

Даже если агент что-то изменил, implementation workspace не должен загрязняться.

## 9.3. Разделить оценки

```text
harness_protocol_result
agent_negotiation_result
implementation_result
```

Пример:

```json
{
  "harness_protocol_result": "violation_detected",
  "agent_decision_score": 0,
  "implementation_evaluated": false
}
```

Это точнее, чем общий “negotiation failed”.

## 9.4. Минимальные реальные сценарии

```text
feature already exists
user misconception
underspecified feature
hidden migration
cross-cutting change
```

### Definition of Done

```text
- preflight работает на snapshot реального workspace;
- любые edits определяются и отбрасываются;
- decision парсится из assistant text;
- oracle answer применяется только после valid clarification;
- negotiation и implementation scores разделены.
```

# Definition of Done этапа R2

R2 завершён, когда:

```text
1. Все proofs получены на clean committed head.
2. Proof records связаны с manifests через SHA-256.
3. README и plan соответствуют реальному CI/local workflow.
4. Lifecycle usage сохраняется либо явно помечается unavailable.
5. Efficiency leaderboard работает только с eligible usage.
6. Smoke score не называется benchmark quality.
7. Requirement adherence считается отдельно.
8. Code-health содержит duplication, complexity и dependency metrics.
9. TodoMVC suite проходит на двух разных корректных реализациях.
10. Намеренно сломанная реализация корректно ловится тестами.
11. Проведён real A/B U3 vs U5 с минимум тремя runs на arm.
12. Отчёт показывает survival, repairs, quality, cost и variance.
```

После этого можно переходить к следующему этапу:

```text
R3 — Cross-Model Lifecycle Benchmark
```

и сравнивать DeepSeek V4 Flash, MiMo V2.5 и Nemotron 3 Ultra уже на достаточно надёжной экспериментальной основе.
