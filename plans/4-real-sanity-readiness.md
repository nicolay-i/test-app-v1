# R1: контур, готовый к проверке с реальным OpenCode

## Исходное состояние

- Исходный коммит: `6385653fb1d26db0fe8485534bb36aae7f2fbf2a`
- Зафиксировано: `2026-07-10T05:47:22Z`
- Проверка типов: пройдена (`pnpm typecheck`)
- Валидация задачи: пройдена (`pnpm bench validate-task tasks/todomvc`)
- OpenCode: доступен (`1.17.15`)
- Адаптер реального OpenCode и проверка жизнеспособности v0: пройдены в запуске
  `20260710T055814Z-real-ec6ced` с `deepseek-v4-flash-free`, U3 и v0.
  Приложение прошло сборку, smoke-проверку в рантайме, базовые E2E- и value-проверки с оценкой 1.00.

## Ограничения до R1

- Артефакты и рабочие каталоги адресовались только логической траекторией, поэтому
  mock- и real-попытки могли использовать один физический workspace.
- Исключение при проверке сгенерированного приложения могло оставить запуск без
  итоговой сводки по версии и траектории. Так произошло в контрольной mock-попытке
  жизненного цикла `r50`: у v0 есть сырые логи OpenCode, но нет сводки об ошибке или траектории.
- JSON-вывод OpenCode сохранялся как лог, но не преобразовывался в структурированный результат событий.

## Критерии готовности

1. У каждой попытки есть неизменяемый идентификатор выполнения и новый workspace по умолчанию.
2. Каждый конечный сценарий формирует сводки по версии и траектории до возврата CLI ненулевого кода.
3. События JSONL дают структурированный ответ ассистента и данные об использовании ресурсов.
4. Mock-сценарии с инъекцией сбоев без учётных данных провайдера доказывают обработку ошибок,
   ремонта, парсера, изоляции и возобновления запуска.
