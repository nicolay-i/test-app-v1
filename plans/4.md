# План следующего этапа: подготовка benchmark-а к честному real OpenCode sanity

Главная цель ближайшего этапа:

> Сделать так, чтобы любой запуск — успешный, упавший, прерванный или отремонтированный — завершался воспроизводимым и пригодным для анализа trajectory, а mock/debug-состояние физически не могло повлиять на real run.

До выполнения этого этапа не стоит расширять матрицу, добавлять новые задачи или улучшать визуальные отчёты.

---

# Milestone R1 — Real-Sanity-Ready Harness

Этап считается завершённым, когда выполнены четыре условия:

1. Каждый запуск получает новый изолированный execution workspace.
2. Любой terminal failure создаёт полный результат, а не исчезает после exception.
3. OpenCode JSONL разбирается в структурированный результат с assistant text и usage.
4. Все критические ветви доказаны deterministic mock/fault-injection тестами.

После этого выполняются:

```text
adapter proof
→ real v0 viability proof
→ real v0..v2 lifecycle proof
→ сравнение двух prompt arms
```

# P0.1. Ввести физическую изоляцию запусков

## Проблема

Сейчас логический `trajectoryId` может совпадать у:

```text
mock и real run;
старого и нового prompt;
старого и нового task definition;
короткого и полного lifecycle;
повторного запуска после изменения scaffold.
```

Если существующий workspace переиспользуется, real агент может начать с уже реализованного mock-приложения.

## Решение

Разделить:

```text
trajectory_id — логическая benchmark cell;
execution_id — конкретная физическая попытка запуска.
```

Предлагаемая структура:

```text
runs/
  <matrix-id>/
    executions/
      <execution-id>/
        execution-manifest.json
        artifacts/
          <trajectory-id>/
            trajectory-summary.json
            v0/
            v1/
        workspaces/
          <trajectory-id>/
```

Пример `execution_id`:

```text
20260710T142233Z-real-7f42c1
```

Для `run-one` создаётся отдельный execution. Для `run-matrix` один execution содержит все trajectories матрицы.

## Execution manifest

```json
{
  "schema_version": "0.2.0",
  "execution_id": "20260710T142233Z-real-7f42c1",
  "run_type": "real",
  "matrix_id": "ape_mvp_001",
  "source_commit": "03abc8f...",
  "config_hash": "...",
  "task_hashes": {
    "todomvc": "..."
  },
  "prompt_hashes": {
    "S2-maintainable-simple": "...",
    "U5-maintainable": "...",
    "E2-smallest-maintainable-change": "..."
  },
  "scaffold_hash": "...",
  "requested_versions": 2,
  "node_version": "...",
  "pnpm_version": "...",
  "opencode_version": "...",
  "started_at": "..."
}
```

## Поведение CLI

По умолчанию:

```text
run-one → всегда новый execution_id;
run-matrix → всегда новый execution_id;
workspace → всегда создаётся с чистого scaffold.
```

Resume только явно:

```bash
pnpm bench run-one --resume <execution-id>
pnpm bench run-matrix --resume <execution-id>
```

Resume разрешается, только если совпадают:

```text
run_type
source commit или разрешённая policy
config hash
task hash
prompt hashes
scaffold hash
requested versions
trajectory definition
```

При несовпадении runner должен остановиться с понятной ошибкой.

Нужны отдельные явные опции:

```text
--fresh                  default
--resume <execution-id>
--force-new-execution
```

Не стоит вводить неявное переиспользование workspace.

## Acceptance criteria

```text
- mock и real с одинаковыми параметрами получают разные execution paths;
- real run никогда не открывает mock workspace;
- изменение prompt приводит к новому execution или resume rejection;
- изменение task.yaml приводит к resume rejection;
- --versions 0 нельзя неявно продолжить как --versions 4;
- старый trajectory-summary сам по себе не считается основанием для resume.
```

# P0.2. Сделать terminal failure полноценным результатом

## Проблема

При OpenCode exit failure runner не должен бросать exception раньше, чем записан полный trajectory result. Иначе failed runs пропадают из aggregation, что систематически улучшает рейтинг нестабильных моделей.

## Решение

Перевести выполнение версии на явную state machine:

```text
prepared
→ agent_running
→ agent_finished | agent_failed
→ evaluation_running
→ evaluation_finished | evaluation_failed
→ repair_running | repair_skipped
→ finalized
```

Ввести единый результат версии:

```ts
type VersionOutcome = {
  versionId: string;
  status: "passed" | "failed" | "incomplete" | "aborted";
  failureClassification:
    | "none"
    | "infra_failure"
    | "model_failure"
    | "opencode_failure"
    | "harness_failure"
    | "unknown";
  failedPhase: string | null;
  score: number | null;
  evaluationCompleted: boolean;
  repairAttempts: number;
  artifactsPath: string;
};
```

## Обязательные артефакты даже при OpenCode failure

```text
vN/
  prompt.md
  run-metadata.json
  opencode.stdout.log
  opencode.stderr.log
  opencode.events.jsonl
  opencode-result.json
  git.diff
  failure-summary.json
  version-summary.json
  report.md
```

На уровне trajectory всегда создаются:

```text
trajectory-summary.json
trajectory-report.md
```

Даже если `v0` не дошёл до build.

## Порядок завершения

```text
1. Зафиксировать failure.
2. Записать version summary.
3. Обновить trajectory summary.
4. Записать report.
5. Закрыть execution metadata.
6. Только затем вернуть ненулевой exit code.
```

Для failure до evaluator:

```json
{
  "status": "failed",
  "failure_classification": "opencode_failure",
  "evaluation_completed": false,
  "score": null
}
```

Не нужно автоматически превращать отсутствие score в `0`. В ranking policy это обрабатывается отдельно.

## Две категории eligibility

Одного `eligible_for_leaderboard` недостаточно.

Добавить:

```json
{
  "eligible_for_reliability_ranking": true,
  "eligible_for_quality_ranking": false
}
```

Логика:

```text
real model/opencode failure:
  reliability ranking — да;
  conditional quality ranking — нет, если evaluator не завершён.

infra/harness failure:
  reliability ranking модели — нет;
  отдельная infra/harness статистика — да.

mock:
  оба ranking — нет.
```

## Acceptance criteria

```text
- OpenCode exit 1 создаёт trajectory-summary.json;
- timeout создаёт terminal result;
- malformed OpenCode stream создаёт terminal result;
- failed v2 остаётся видимым после успешных v0/v1;
- aggregation не теряет failed trajectories;
- CLI возвращает exit code 1 только после записи всех artifacts.
```

# P0.3. Реализовать нормальный parser OpenCode event stream

## Проблема

`opencode run --format json` возвращает поток JSON events, а не один финальный JSON-объект. Нельзя парсить весь stdout как один объект.

## Решение

Добавить отдельный модуль:

```text
packages/runner/src/opencodeEventParser.ts
```

Контракт:

```ts
type OpenCodeUsage = {
  inputTokens: number | null;
  outputTokens: number | null;
  cacheReadTokens: number | null;
  cacheWriteTokens: number | null;
  totalTokens: number | null;
  reportedCost: number | null;
  source: "events" | "stats" | "unavailable";
  complete: boolean;
};

type ParsedOpenCodeResult = {
  sessionId: string | null;
  assistantText: string;
  finishStatus: string | null;
  usage: OpenCodeUsage;
  toolErrors: string[];
  malformedLines: number;
  unknownEventTypes: string[];
  streamStatus: "complete" | "partial" | "invalid";
};
```

Parser должен:

```text
1. читать stdout построчно;
2. JSON.parse каждую строку отдельно;
3. сохранять malformed lines, не теряя весь stream;
4. собирать assistant text в правильном порядке;
5. извлекать session id;
6. извлекать finish status;
7. извлекать usage/cost, если они есть;
8. сохранять неизвестные event types;
9. возвращать partial result при неполном stream.
```

Артефакты:

```text
opencode.events.jsonl   — оригинал;
opencode-result.json    — parsed result;
assistant-response.md   — собранный текст ответа.
```

Negotiation должен парсить decision только из:

```text
parsedResult.assistantText
```

а не из raw stdout.

## Usage не должен быть блокирующим

Если провайдер или OpenCode не возвращает токены:

```json
{
  "totalTokens": null,
  "reportedCost": null,
  "source": "unavailable",
  "complete": false
}
```

Отсутствие usage должно быть явно видно, а не трактоваться как `0`.

## Acceptance criteria

Подготовить fixtures:

```text
complete stream;
multi-part text;
tool calls между text events;
malformed line;
truncated stream;
no text;
usage present;
usage absent;
multiple JSON objects.
```

Для каждого fixture parser должен выдавать предсказуемый результат.

# P0.4. Добавить deterministic fault injection для mock

## Цель

Mock должен проверять не только happy path. Иначе repair, terminal failure и resume остаются недоказанными до дорогого real run.

## Интерфейс

Например:

```bash
--mock-profile happy
--mock-profile opencode-fail-v1
--mock-profile build-fail-v2-repair-success
--mock-profile e2e-fail-v2-repair-fail
--mock-profile malformed-events
--mock-profile timeout-v0
```

Или через YAML:

```yaml
mock:
  failures:
    - version: v1
      phase: opencode
      type: exit_nonzero

    - version: v2
      phase: e2e
      type: assertion_failure

  repairs:
    v2: success
```

## Обязательные fault scenarios

### F1. OpenCode failure на v0

Проверяет:

```text
terminal result;
failure classification;
trajectory visibility;
non-zero CLI exit.
```

### F2. OpenCode failure на v2

Проверяет:

```text
v0/v1 не теряются;
first_failed_version = v2;
aggregation видит partial survival.
```

### F3. Build failure + repair success

Проверяет:

```text
repair-1 artifacts;
repair cost;
final version status;
original failure не перезаписан.
```

### F4. E2E failure + repair failure

Проверяет:

```text
две попытки видны отдельно;
trajectory считается failed;
repair_attempts = 1.
```

### F5. Timeout

Проверяет:

```text
process termination;
classification;
partial logs;
terminal summary.
```

### F6. Malformed event stream

Проверяет:

```text
raw events сохранены;
parser не роняет весь harness;
streamStatus = partial/invalid.
```

### F7. Mock → real isolation

```text
1. Запустить mock с конкретной cell.
2. Запустить real/fake-real с той же cell.
3. Проверить, что workspace paths различаются.
4. Проверить отсутствие файлов mock-приложения в real workspace.
```

### F8. Stale resume

```text
1. Начать execution.
2. Изменить prompt/task hash.
3. Попытаться resume.
4. Получить deterministic rejection.
```

## Acceptance criteria

Все эти сценарии должны запускаться без сети и стать частью CI.

# P0.6. Провести fairness audit TodoMVC tests

До real lifecycle важно убедиться, что тесты проверяют продукт, а не конкретную DOM-реализацию.

## Для каждого теста зафиксировать

```text
acceptance criterion;
observable user behavior;
selector strategy;
почему тест не зависит от внутренней архитектуры;
допустимые альтернативные реализации.
```

## Запрещённые предположения без явного контракта

```text
точный CSS class;
конкретный component name;
конкретный localStorage key;
конкретная вложенность li/div;
фиксированный внутренний state shape;
обязательная библиотека;
конкретная реализация drag/drop или filtering.
```

Пример: требование “overdue todo визуально отличается” не должно автоматически означать обязательный класс `.overdue`, если это не часть task contract.

```text
видимый overdue label;
aria-label;
достаточно явный computed style;
отдельный visual/manual check.
```

## Harness fixtures

Полезно иметь минимум:

```text
compliant-reference-A;
compliant-alternative-DOM-B;
known-broken-C.
```

Тесты должны:

```text
проходить на A;
проходить на B;
падать на C.
```

Это лучший способ выявить implementation-specific harness до запуска моделей.

# P1. Обновить aggregation под lifecycle semantics

Текущий главный вопрос:

> Какой профиль даёт лучший lifecycle quality на единицу lifecycle cost?

Поэтому одного leaderboard по final score недостаточно.

## Разделить отчёты

### Reliability leaderboard

Основной порядок:

```text
1. survival rate;
2. survived versions;
3. regression-free rate;
4. repair-free rate;
5. first failed version.
```

### Conditional quality leaderboard

Сравнивает качество только тех версий, до которых trajectory дошёл и которые были полноценно evaluated:

```text
mean evaluated quality;
final evaluated quality;
minimum version quality;
quality degradation slope.
```

### Efficiency leaderboard

Когда usage доступен:

```text
successful versions / total tokens;
quality-adjusted survival / total tokens;
repair tokens / total tokens;
tokens per passing version;
```

## Failed real trajectories

Они не должны исчезать.

Пример:

```text
OpenCode failure на v0
→ reliability failure;
→ quality score unavailable;
→ lifecycle cost сохраняется, если usage доступен.
```

## Infra/harness failures

Показывать отдельно:

```text
infra failure rate by provider/model;
harness failure rate by task/test;
excluded from model ranking by default.
```

## Новые поля trajectory summary

```json
{
  "survival_rate": 0.5,
  "regression_free_versions": 1,
  "repair_free_versions": 1,
  "quality_degradation_slope": null,
  "lifecycle_tokens": null,
  "tokens_per_passing_version": null,
  "usage_complete": false
}
```

# P1. Первый real OpenCode sanity

Real sanity нужно разбить на отдельные доказательства. Не начинать сразу с двух prompts и `v0..v4`.

## Шаг A. Environment/adapter doctor

Добавить или использовать команду:

```bash
pnpm bench doctor
```

Проверки:

```text
Node version;
pnpm version;
git;
OpenCode binary;
OpenCode version;
конфигурация provider;
доступность выбранного model id;
Playwright browser;
write access к run directory;
достаточное свободное место.
```

Не сохранять секреты. Сохранять только имена переменных и факт наличия.

## Шаг B. Adapter proof

Один запуск:

```text
task: todomvc
model: один configured model
user prompt: U3 или текущий baseline
versions: 0
repair: disabled
fresh execution
```

Пример:

```bash
pnpm bench run-one \
  --task todomvc \
  --model deepseek-v4-flash-free \
  --system S2-maintainable-simple \
  --user U3-semantic-ui \
  --edit E2-smallest-maintainable-change \
  --versions 0 \
  --run-type real \
  --fresh
```

Adapter proof считается состоявшимся, даже если модель не построила приложение, при условии что:

```text
OpenCode реально вызван;
events распарсены;
terminal trajectory сформирован;
failure классифицирован;
artifacts полные;
verify-run проходит.
```

## Шаг C. Task viability proof

Нужен хотя бы один real `v0`, который:

```text
собирается;
открывается;
проходит critical base e2e;
проходит value checks;
не имеет критических runtime errors.
```

Если первый v0 не прошёл, не надо вручную исправлять generated app. Нужно сохранить результат и решить:

```text
это model failure;
prompt issue;
task ambiguity;
harness issue;
scaffold issue.
```

## Шаг D. Short lifecycle proof

После успешного v0:

```text
v0 → v1 due dates → v2 search
```

То есть:

```bash
--versions 2
```

Проверить:

```text
cumulative tests;
version diffs;
usage;
repair;
trajectory summary;
aggregation.
```

## Шаг E. Первый A/B

Только после working `v0..v2`:

```text
1 model
2 user prompt arms
1 system prompt
1 edit prompt
1 run per arm
v0..v2
```

Например:

```text
U3-semantic-ui
vs
U5-maintainable
```

Это всё ещё sanity, а не статистически надёжный вывод.

# P1. Real negotiation после исправления parser

Negotiation не стоит запускать до появления корректного JSONL parser.

## Preflight должен работать на реальном workspace

Вместо общего текста про TodoMVC агент должен получить доступ к workspace конкретной версии.

Контекст:

```text
current version;
applied evolution steps;
current acceptance criteria;
known features;
current test inventory;
compact file tree;
known failures.
```

## Защита от изменения файлов

Preflight запускается на отдельной ephemeral копии workspace:

```text
1. Сделать snapshot commit/hash.
2. Запустить negotiation preflight.
3. Проверить git diff.
4. Любое изменение файлов отметить как protocol violation.
5. Не переносить эти изменения в implementation workspace.
```

Можно дополнительно делать копию read-only, но git diff check всё равно нужен.

## Full negotiation flow

```text
preflight
→ decision score
→ oracle response
→ новый writable implementation workspace
→ implementation
→ обычный lifecycle evaluator
```

Отдельно считать:

```text
decision accuracy;
question recall;
unnecessary question penalty;
existing feature detection;
protocol violation;
final implementation quality.
```

# P2. Jury workflow после real proof

Текущий jury skeleton можно сохранить. После real proof добавить два режима blind:

```text
light blind:
  скрыты model/prompt/run ids;
  task и automated checks видны.

strict blind:
  дополнительно скрыты run_type, raw scores и failure labels;
  reviewer видит requirements, app, screenshots, diff и code.
```

Для pairwise review variants должны называться:

```text
Variant A
Variant B
```

а mapping храниться вне packet.

# Рекомендуемый порядок коммитов

## Commit 1 — execution identity

```text
execution_id;
execution manifest;
новая структура directories;
fresh workspace default;
strict resume validation.
```

## Commit 2 — terminal failure finalization

```text
VersionOutcome;
нет throw до finalization;
summary/report для OpenCode failure;
score:null semantics;
две категории ranking eligibility.
```

# Definition of Done текущего цикла

Цикл завершён, когда выполнены следующие проверки.

## 1. Isolation proof

```text
mock и real одной cell имеют разные workspaces;
stale resume отклоняется;
новый run всегда стартует с чистого scaffold.
```

## 2. Failure proof

```text
OpenCode failure на v0 создаёт полный terminal trajectory;
OpenCode failure на v2 сохраняет v0/v1 и first_failed_version;
aggregation видит оба результата.
```

## 3. Parser proof

```text
JSONL fixtures проходят;
assistant text извлекается;
negotiation decision парсится из assistant text;
malformed stream не уничтожает artifacts.
```

## 4. Repair proof

```text
build/e2e failure запускает repair;
repair success и repair failure сохраняются отдельно;
repair cost не теряется.
```

## 5. Mock proof

```bash
pnpm bench run-one ... --versions 4 --run-type mock
pnpm bench verify-run --execution <id>
pnpm bench aggregate --execution <id>
```

Mock не попадает в model leaderboard.

## 6. Real proof

Есть три отдельных результата:

```text
real adapter proof — полный terminal trajectory;
real v0 viability proof — passed base app;
real v0..v2 lifecycle proof — cumulative tests и summary.
```

Только после этого можно запускать сравнение двух prompt arms и считать harness готовым к первой ограниченной benchmark-матрице.
