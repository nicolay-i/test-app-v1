# План реализации раздела «Страны»

## Цель

Добавить публичный раздел **«Страны»**, заменить им пункт **«Районы / Areas»** в основном меню и организовать переход:

```text
Страны → Районы выбранной страны → Каталог объектов района
```

Основной пользовательский маршрут:

```text
/{locale}/countries
/{locale}/areas?country={countryId}
/{locale}/catalog?area={areaId}
```

Маршрут `/areas` сохраняется и продолжает работать, но больше не отображается в основном меню.

---

# 1. Зафиксировать правила поведения

## Требования

1. В основном desktop- и mobile-меню:

   * добавить пункт «Страны»;
   * удалить пункт «Районы».

2. Страница стран:

   * показывает только активные публичные страны;
   * страна должна иметь хотя бы один опубликованный доступный объект;
   * нажатие на страну открывает районы этой страны.

3. Страница районов:

   * `/areas` показывает все районы;
   * `/areas?country=egypt` показывает только районы Египта;
   * параметр `country` остаётся в URL;
   * неизвестный `countryId` не должен приводить к отображению всех районов.

4. Нажатие на район:

   * ведёт в `/catalog?area={areaId}`;
   * существующее поведение сохраняется.

5. Административные страницы стран и районов не меняются, кроме возможного расширения тестов.

## Проверка

* Согласован URL-формат `?country={countryId}`.
* Согласовано, что в MVP у стран нет отдельных hero-изображений.
* `/areas` не удаляется из приложения.

3. `countryId` неизвестен — показать `NotFound` или отдельное состояние «Страна не найдена».

# 18. Проверить статическую генерацию и регенерацию

## Проблема

Публичность страны зависит от наличия опубликованных и доступных объектов.

Следовательно, страницу `/countries` нужно регенерировать при:

* создании страны;
* изменении страны;
* активации/деактивации страны;
* публикации объекта;
* снятии объекта с публикации;
* переводе последнего объекта страны в `sold`;
* переносе объекта в район другой страны;
* изменении района или города.

## Изменения

Добавить зависимость страницы:

```text
marketing:countries
countries
areas
properties
```

Обновить mapping entity changes → affected static pages.

## Проверка

Сценарий:

1. У страны один опубликованный объект.
2. Страна видна на `/countries`.
3. Объект снимается с публикации.
4. Static generation выполняется.
5. Страна исчезает со страницы.

# Рекомендуемое разбиение по subagents

## Subagent 1 — Frontend routing и navigation

Задачи:

* `App.tsx`;
* `routes.ts`;
* Header;
* Footer;
* локализации меню.

Проверка:

* маршруты;
* desktop/mobile navigation;
* ссылки.

## Subagent 2 — CountriesPage и AreasPage

Задачи:

* новая страница стран;
* фильтрация районов;
* URL-state;
* country context;
* loading/error/empty states.

Проверка:

* component tests;
* ручные сценарии фильтрации.

## Subagent 3 — Static payload frontend

Задачи:

* типы payload;
* loaders;
* matchers;
* cache;
* guards;
* prefetch.

Проверка:

* tests static payload;
* fallback API.

## Subagent 4 — Backend static generation

Задачи:

* route resolution;
* marketing page resolver;
* client payload;
* SEO;
* regeneration dependencies.

Проверка:

* API/static generation tests;
* прямое открытие HTML.

## Subagent 5 — E2E и regression

Задачи:

* обновить существующие ожидания меню;
* добавить Countries → Areas → Catalog;
* проверить mobile и reload;
* запустить полный test suite.

Проверка:

* отчёт по каждой acceptance criterion;
* список обнаруженных регрессий.

Для первого этапа разумно оставить карточки стран без загружаемых изображений: флаг, название и переход. Это заметно сокращает объём и не требует миграции базы.
